2-Week Data Modeling Plan

Focus: Supabase + PostgreSQL | Time: 60-90 mins/day | Goal: Build strong, practical schema design skills

Learning Goal

By the end of 2 weeks, you should be able to:

  • Choose between current-state tables and history/log tables with clear reasoning.
  • Use constraints and foreign keys to protect data quality.
  • Design Row Level Security (RLS) based on trust boundaries.
  • Write and review migrations with confidence in real project work.

Week 1: Core Modeling Skills

Day 1: Table Design Fundamentals
  • Learn entities, relationships, primary keys, and foreign keys.
  • Read: PostgreSQL DDL, PostgreSQL Constraints
  • Practice: sketch users, codewars_accounts, and one related table.
Day 2: Constraints as Safety Nets
  • Study NOT NULL, UNIQUE, CHECK, and FK cascade behavior.
  • Practice writing and testing constraints for usernames and ownership.
Day 3: Normalization (Practical)
  • Learn 1NF/2NF/3NF at a practical level.
  • Read: Database Normalization
  • Review one migration and identify any redundant or mixed concerns.
Day 4: RLS and Trust Boundaries
  • Understand who can read/write which data and why.
  • Read: Supabase RLS Guide
  • Practice: separate user-managed fields from system-managed fields.
Day 5: UPDATE vs Log Table Modeling
  • Compare update-in-place vs append-only history designs.
  • Read: Temporal Patterns
  • Practice both designs for Codewars progress data.
Day 6: Review and Notes
  • Create a one-page summary: when to use snapshot tables, when to use log tables.
Day 7: Teach-Back Check
  • Explain your understanding out loud in 5 minutes without notes.

Week 2: Project-Style Practice

Day 8: Read Existing Migrations
  • Inspect migration files and summarize design choices in 2-3 lines each.
Day 9: Model One Feature Two Ways
  • Build one schema with UPDATE-in-place and another with log history.
  • Compare complexity, query simplicity, and analytical value.
Day 10: Query Patterns
  • Write current-state query, latest-history query, and trend-over-time query.
Day 11: Migration Evolution Plan
  • Plan a safe transition from snapshot-only to hybrid snapshot + log.
  • Think in steps: add table, backfill, switch reads, deprecate old paths.
Day 12: Security-First Modeling
  • Write policy intent before SQL: who can insert, update, delete, select.
Day 13: PR Review Simulation
  • Review your own schema as a mentor would: history value, trust boundaries, future queries.
Day 14: Final Decision Template
  • Create a reusable checklist:
    • Data shape
    • Constraints
    • RLS
    • Snapshot vs log decision
    • Future migration path

Success Criteria

  • You can justify schema choices clearly in PR discussions.
  • You can explain UPDATE vs log-table trade-offs with confidence.
  • You design RLS alongside schema, not as an afterthought.
  • You can propose safe evolution paths for future requirements.
